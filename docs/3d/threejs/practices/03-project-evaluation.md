## 适用场景

| 场景 | 怎么用本文 |
|------|------------|
| 作品集写 README | 按 [评分表](#评分表模板) 填数据，附截图与 Perf 面板 |
| 团队 POC 评审 | 六维度打分，低于 60 不进迭代 |
| 上线前验收 | 对照 [工程 06](../engineering/06-quality-and-delivery.md) + 本文性能/稳定性 |
| 面试讲项目 | 用「维度 + 数字 + 取舍」结构，而不是只演示 orbit |

**评估原则**：3D 项目没有单一「满分标准」，**先定项目形态，再定权重**——产品展示重视觉，孪生重稳定与数据，大屏重联动与帧率档位。

---

## 评估三阶段

```
POC（能跑）  →  迭代（能留）  →  验收（能交）
   ↓              ↓                ↓
 功能闭环      性能/架构达标      监控/文档/兼容
```

| 阶段 | 最低门槛 | 典型产出 |
|------|----------|----------|
| **POC** | 模型能加载、相机可控、无 blocking 报错 | 录屏 + 已知问题列表 |
| **迭代** | 性能预算达标、dispose 完整、错误有 UI | Perf 数据 + 架构图 |
| **验收** | 弱网/移动端/多浏览器通过、CI 与清单绿 | 验收报告 + 运维 playbook |

不要在 POC 阶段用生产环境的全部指标卡死自己；但 **POC 就要记录三角面、包体、首屏秒数**，否则后面无法对比优化效果。

## 六大维度

| 维度 | 评什么 | 主要证据 |
|------|--------|----------|
| **性能** | 加载、帧率、内存、draw call | Perf 面板、Network、Performance、Memory |
| **视觉** | PBR、光照、后处理、资产质量 | 固定视角截图、美术验收 |
| **交互** | 拾取、标注、运镜、与 DOM 协作 | 操作录屏、点击链路说明 |
| **工程** | 分层、dispose、错误处理、可测试 | 目录结构、Code Review |
| **稳定** | 弱网、404、路由切换、context lost | 手工 + E2E 用例 |
| **可维护** | 文档、命名、资源管线、版本统一 | README、glTF 规范、monorepo |

**总分建议**：每维度 0～100，按 [形态权重](#按项目形态调整权重) 加权；**加权分 ≥ 75** 可对外演示，**≥ 85** 可写进简历核心项目。

## 按项目形态调整权重

| 维度 | 产品展示 P1 | 数字孪生 P2 | 大屏 3D P3 | 配置器 P4 |
|------|:-----------:|:-----------:|:------------:|:---------:|
| 性能 | 20% | 25% | 25% | 20% |
| 视觉 | **30%** | 15% | 10% | **25%** |
| 交互 | 20% | **25%** | **25%** | 20% |
| 工程 | 15% | **20%** | 20% | 15% |
| 稳定 | 10% | 15% | 15% | 10% |
| 可维护 | 5% | 10% | 5% | 10% |

权重随甲方要求调整；例如政府孪生大屏往往把 **稳定 + 性能** 拉到 40% 以上。

## 性能：指标与测量

### 推荐预算（WebGL / Three.js）

与 [工程 06 · 性能预算](../engineering/06-quality-and-delivery.md) 对齐：

| 指标 | 桌面（参考） | 移动端（参考） | 怎么量 |
|------|-------------|---------------|--------|
| 首屏可交互（含 3D） | < 3s | < 5s | Network 4G 节流 + `loadMs` 打点 |
| 稳态 FPS | ≥ 55 | ≥ 28 | [Perf 面板](../../../apps/threejs-demos/README.md) / Stats |
| 单场景 glb 总量 | < 15MB | < 8MB | 构建脚本 / CI |
| 可见三角面 | < 50 万 | < 20 万 | `renderer.info.render.triangles` |
| Draw calls | < 200 | < 80 | Perf 面板 **Draw** |
| DPR | cap 2 | cap 1.5 | `renderer.getPixelRatio()` |
| 路由离开内存 | 回落 | 回落 | DevTools Memory，重复进入 10 次 |

### 测量步骤（固定流程，结果才可对比）

1. **环境**：Chrome 最新版；移动端用真机或 DevTools 设备模拟 + **4G 节流**各测一次。
2. **场景**：固定相机位置、固定模型集、关 autoRotate（测 FPS 时）。
3. **工具**：
   - 本仓库 Demo：canvas 左下角 **Perf**（[threejs-demos](../../../apps/threejs-demos/README.md)）
   - Chrome **Performance**：看 GPU、Long Task
   - Chrome **Memory**：load/unload 循环
   - **Spector.js**（可选）：draw call 明细
4. **记录模板**：

```markdown
| 环境 | 首屏 loadMs | FPS | 三角面 | Draw | DPR | 备注 |
|------|-------------|-----|--------|------|-----|------|
| MBP Chrome | 2100 | 60 | 82k | 45 | 2 | 三点光 |
| Android 4G | 4800 | 32 | 82k | 45 | 1.5 | 关 Bloom |
```

### 性能评分参考（单维度 100 分）

| 得分 | 条件（桌面 FPS / 首屏） |
|------|-------------------------|
| 90+ | FPS ≥ 58 且首屏 < 2.5s |
| 75～89 | FPS ≥ 50 且首屏 < 3.5s |
| 60～74 | FPS ≥ 30，可用但有明显档位妥协 |
| < 60 | 长期 < 30fps 或首屏 > 8s 无 loading |

优化顺序（性价比）：**减 draw call / 三角面 → 压缩纹理与 Draco → 分区加载 → 关 shadow/Bloom 档位 → 按需渲染**（见 [09 性能](../09-performance-and-dispose.md)、[14 大场景](../advanced/14-large-scene-loading.md)）。

## 视觉与资产

| 检查项 | 过关标准 |
|--------|----------|
| 模型 | glTF 2.0；无破面、法线正确；命名规范（`zone-a`、`device-pump-01`） |
| PBR | 产品类：Standard + HDRI 或三点光；金属/粗糙度合理 |
| 贴图 | 2 的幂次；KTX2/Basis 优先；单张 < 2048（移动端） |
| 光照 | 无死黑、无爆白；阴影 intentional，非默认糊一片 |
| 后处理 | Bloom/Outline 有移动端关闭策略（[12 后处理](../advanced/12-post-processing.md)） |
| 一致性 | 与大屏/ECharts 色板、UI 主题统一（P3） |

**证据**：固定 3 张截图（全貌 / 特写 / 告警或高亮态）+ gltf-validator 报告。

## 交互与体验

| 检查项 | 过关标准 |
|--------|----------|
| 相机 | Orbit 有 min/max distance；产品类限制仰角；双击 fit |
| 拾取 | 点击有反馈（高亮 / Outline）；cursor 变化 |
| 标注 | CSS2D/DOM 不严重遮挡；缩放时可读（[15 标注](../advanced/15-html-labels.md)） |
| 加载 | 进度条或 skeleton；失败可重试，不白屏 |
| 联动 | 3D ↔ 图表 / 面板双向至少一条完整链路（P2/P3） |
| 无障碍 | 关键操作有 DOM 入口（纯 canvas 操作需说明） |

**评分**：核心路径 30 秒内能讲完 → 80 分；需要口头解释才能操作 → 60 分以下。

## 工程与架构

对照 [工程 01](../engineering/01-why-not-raw-threejs.md)、[03 目录](../engineering/03-project-structure.md)：

| 检查项 | 过关标准 |
|--------|----------|
| 分层 | 业务不 `import three`；引擎层统一 load/dispose |
| 生命周期 | `onMounted` 创建 / `onUnmounted` destroy；无重复 canvas |
| 资源 | Loader 封装；404 降级；Draco 路径可配置 |
| 状态 | 设备 id → mesh 映射清晰；数据驱动颜色/标签 |
| 测试 | 至少：normalize、dispose、registry 单测；可选 Playwright |
| 依赖 | three 版本锁定；monorepo 多项目一致 |

**反模式（扣分）**：业务里到处 `scene.add`；切换模型不 dispose；全局单例 Renderer 不销毁。

## 稳定性与兼容

| 检查项 | 过关标准 |
|--------|----------|
| 弱网 | 4G 节流下可完成加载或明确失败提示 |
| 路由 | SPA 切换 10 次无 context 泄漏、无叠 canvas |
| 浏览器 | Chrome + Safari（iOS 必测）；WebGL 不支持时有降级文案 |
| Context lost | 监听 `webglcontextlost`；必要时提示刷新 |
| 长时间运行 | 大屏 8h 无内存单调涨（孪生/Data 推送场景） |

## 可维护与协作

| 检查项 | 过关标准 |
|--------|----------|
| README | 在线地址、架构图、性能表、本地启动命令 |
| 资源管线 | 美术交付规范 → 导出脚本 → CDN（[工程 04](../engineering/04-asset-pipeline.md)） |
| 变更 | 换模型不改业务代码；配置 JSON 可驱动场景（可选） |
| 开源对照 | 能说明为何自研 viewer 而非 model-viewer / xeokit（[工程 07](../engineering/07-open-source-reference.md)） |

## 一票否决项

任一命中，**不应上线**（POC 可例外但须书面记录）：

- 路由/leaving 页面后 WebGL 内存持续增长
- 生产环境模型 404 导致白屏或未捕获异常
- 移动端长期 < 20fps 且无降级档位
- 无 loading、无 error UI，用户只能看空白 canvas
- 业务层直接操作 Scene，导致无法测试与复用
- 硬编码生产 CDN 密钥或内网地址进仓库

## 作品集 vs 生产项目

| 项 | 作品集（求职） | 生产项目 |
|----|----------------|----------|
| 性能 | 桌面数据 + 1 组移动端抽样即可 | 全档位 + 监控上报 |
| 视觉 | 1 个高质量模型胜过 10 个糙模 | 美术流程 + validator CI |
| 工程 | 分层清晰 > 功能堆砌 | 单测 + E2E + Code Review |
| 文档 | README 含性能表 + 30s 录屏 | 运维 playbook + 值班指标 |
| 通过线 | 加权 **≥ 75**，P1+(P2\|P3) | **≥ 85** + [工程 06 清单](../engineering/06-quality-and-delivery.md) |

作品集评估实例：用 [threejs-demos/portfolio](../../../apps/threejs-demos/README.md) 跑 P1，展开 Perf 填表，对照 [02 作品集验收](./02-portfolio-projects.md#上线自检)。
