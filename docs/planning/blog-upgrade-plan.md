# 博客升级规划

> 目标：在保留现有内容与能力的前提下，让博客更精美、现代、好维护。  
> **包管理**：本项目使用 **pnpm**，本地与 CI 均不使用 yarn。

## 一、现状概览

| 项目 | 当前 |
|------|------|
| 框架 | VuePress 1.x + vuepress-theme-reco 1.6.17 |
| 包管理 | pnpm（packageManager: pnpm@9.x） |
| 文章量 | 约 331 篇 Markdown |
| 部署 | GitHub Actions → gh-pages（pnpm install + pnpm docs:build），域名 blog.zkkysqs.top |
| 评论 | Valine |
| 其它 | 看板娘、Meting 音乐、PWA、GA、百度推送、拼音 URL、代码复制等 |

主题 reco 1.x 已较久未大更新，视觉与交互有提升空间；VuePress 1 基于 Webpack，构建偏慢。

---

## 二、升级路线（三选一或组合）

### 方案 A：在现有栈上「美化」——改动最小

**思路**：不换框架和主题，只做样式与配置升级，让 reco 看起来更精致。

**可做事项**：

1. **视觉**
   - 统一设计语言：在 `palette.styl` / `index.styl` 里定一套主色、字重、圆角、阴影（如卡片、按钮、代码块）。
   - 字体：为正文/标题选用 1～2 款 Web 字体（如霞鹜文楷 + 系统无衬线），提升排版观感。
   - 首页/列表：通过覆盖 reco 的 class，优化卡片间距、悬停态、摘要字数与行高。
   - 代码块：已有机型装饰，可再统一背景、行高、边距，使「窗口感」更统一。

2. **交互与体验**
   - 深色模式：若 reco 支持，在 themeConfig 中开启并统一配色。
   - 滚动与动效：smoothScroll 已开，可适度增加锚点滚动、回到顶部按钮样式。
   - 移动端：检查导航、侧栏、代码横向滚动在窄屏下的表现并微调。

3. **内容与 SEO**
   - 为重要文章补全 frontmatter：`description`、`tags`，利于摘要与搜索。
   - 确认 sitemap、百度推送、GA 在升级后仍正常。

**工作量**：小（约 1～3 天集中改样式 + 逐页抽查）。  
**风险**：低，不涉及迁移。  
**适合**：优先要「更好看」、不想动文章和部署时选。

---

### 方案 B：迁移到 VuePress 2 + theme-reco 2.x

**思路**：用官方 VuePress 2 与新版 theme-reco，获得更好性能与更多主题能力，再在此基础上做美化。

**变化概要**：

- VuePress 2：Vite 构建、Vue 3、默认更快的 dev/build。
- theme-reco 2.x：新布局与组件，需对照 [官方文档](https://theme-reco.vuejs.press/) 做配置与样式覆盖。
- 插件：部分 VP1 插件需替换为 VP2 等价物或官方推荐（如搜索、PWA、评论等）。

**必做事项**：

1. 在**新分支**建 VP2 项目，配置 `theme-reco`、导航、侧栏、frontmatter 兼容（date、categories、tags、sticky 等）。
2. 插件迁移：Valine → 可继续用或换 Waline；Meting、看板娘等查 VP2 兼容或替代。
3. 批量检查 331 篇 md：frontmatter、内部链接、图片路径、扩展语法（如 PlantUML）在 VP2 下是否正常。
4. 样式迁移：把现有 `index.styl` / `palette.styl` 的修改在新主题的 CSS 变量/覆盖文件中复现。
5. CI：保持使用 `pnpm install` + `pnpm docs:build`，确认 CNAME、部署目录不变。

**工作量**：中（约 1～2 周，含回归测试）。  
**风险**：中，需逐类检查插件与 frontmatter。  
**适合**：希望长期用 Vue 系、愿意做一次性迁移时选。

---

### 方案 C：换栈（如 VitePress / VuePress 3 / 其它）

**思路**：若更看重「极简 + 速度」或「强定制」，可考虑 VitePress、VuePress 3 等，但需接受主题与插件生态差异。

- **VitePress**：文档向、默认主题简洁，博客需自己加「首页列表、分类、时间线」或选社区主题，评论/音乐等要自行集成。
- **VuePress 3**：仍在演进，可观望后再决定。
- **其它（如 Hexo、Astro 等）**：内容需迁移，frontmatter 与 URL 要批量处理，工作量最大。

**工作量**：大（2 周以上）。  
**风险**：高，涉及内容与 URL 迁移、评论/统计等重配。  
**适合**：有明确「换生态」需求时再考虑。

---

## 三、推荐执行顺序

1. **先做方案 A（在现有栈上美化）**
   - 快速见效，风险低，便于你确认「更精美」的方向（配色、字体、留白、动效等）。
   - 为后续若选 B/C 积累一份「目标视觉」清单，迁移时可直接在新主题里复现。

2. **再评估是否做方案 B**
   - 若 A 做完仍觉得主题天花板明显（性能、功能、可定制性），再在**新分支**做 VP2 + reco 2 迁移，把 A 的样式结论带过去。

3. **方案 C 仅在有强烈需求时考虑**
   - 例如必须 VitePress 的默认体验、或要 Astro 等多框架能力时再投入。

---

## 四、方案 A 可落地的第一步（建议本周内）

在不改框架的前提下，优先做这几项，整体观感会有明显提升：

| 步骤 | 内容 | 涉及文件/位置 |
|------|------|----------------|
| 1 | 定主色与字体 | `docs/.vuepress/styles/palette.styl`，必要时 `headConfig` 加字体 link |
| 2 | 统一代码块与卡片样式 | `docs/.vuepress/styles/index.styl` |
| 3 | 首页/列表卡片与摘要 | 同上，覆盖 reco 的 `.abstract-item` 等 |
| 4 | 深色模式（若主题支持） | `themeConfig.js` 或主题文档 |
| 5 | 在本地与移动端各点几条链接做一次视觉与链接检查 | 无代码，仅验收 |

如需，我可以按你选的方案（例如先做 A 的第一步）给出具体修改示例（如 `palette.styl` / `index.styl` 的片段和注释），直接可贴进项目使用。
